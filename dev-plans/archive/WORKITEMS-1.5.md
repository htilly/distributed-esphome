## Rebrand: ESPHome Fleet

The product has grown beyond its original "distribute ESPHome compiles to faster machines" identity ŌĆö 1.4 added fleet-scale management (version pinning, scheduled upgrades, device tags, Schedules tab). The current name "Distributed ESPHome" overweights the compile mechanism and underweights the actual user value. Rebrand to **"ESPHome Fleet"**.

**Ordering:** This section executes **before** the HA Native Integration section below. The integration's `DOMAIN` determines service names, entity IDs, events, and config entries ŌĆö all of which are scripted against by users in automations. Changing the domain post-launch breaks every user's config. Doing the rebrand first lets us ship the integration with the `esphome_fleet` domain from day one.

**Scope: user-facing strings + integration domain.** Since HI.* ships in the same release with no users yet, we can also rename the in-progress integration domain (`distributed_esphome` ŌåÆ `esphome_fleet`) at zero migration cost. The rest stays the same ŌĆö no breaking changes for existing add-on users.

**What stays the same:** repo name (`weirded/distributed-esphome`), Docker image names (`esphome-dist-server`, `esphome-dist-client`), add-on slug (`esphome_dist_server`), Python module names, logger names, and the YAML comment marker (`# distributed-esphome:` ŌĆö parsed from users' existing YAML files; changing it would be a user-data migration).

## HA Native Integration

Custom integration that makes ESPHome Fleet a first-class HA citizen: native services callable from automations, `update` entities on the HA dashboard, and zero-config discovery via mDNS. ## HA User Authentication

ESPHome Builder (the stock dashboard) has `auth_api: true` in its add-on `config.yaml`. We don't. That single flag gives an add-on access to Supervisor's `/auth` endpoint, which lets us actually *identify* HA users (not just "they passed HA Ingress") and validate their credentials on direct-port access.

Currently, `/ui/api/*` is unauthenticated by design ŌĆö we trust HA Ingress (F-03 in the security audit is explicitly WONTFIX for this reason). With `auth_api`, F-03 becomes a real fix rather than an accepted risk: direct-port `:8765` can require a valid HA token, and we can attribute every mutation to the HA user who initiated it.

Scope: additive. Ingress-trusted access keeps working unchanged. Direct-port access gains an auth requirement. Action attribution surfaces in the server log + queue entries. **No new login UI** ŌĆö for Ingress, the user is already logged into HA; for direct-port, we return 401 + a clean `WWW-Authenticate` challenge and document that direct-port clients must send `Authorization: Bearer <HA-long-lived-access-token>`.

## Server ESPHome Unbundling

Bugs #51 (reverted) and #56 (PyObjC leak, twice) are symptoms of the same root: the server bakes a pinned `esphome==X.Y.Z` into its Docker image, so every version bump re-resolves a huge transitive graph with platform-conditional sharp edges. Bug #20 (ESPHome 2026.4.0 API break) is a sibling ŌĆö the pinned server was blind to the upstream API change until we tried to bump. Patching each class one at a time is losing ground.

Full unbundling ŌĆö ship the server with **no** ESPHome baked in, install the right version at first boot into `/data/esphome-versions/<ver>/` via the existing `VersionManager` (`ha-addon/client/version_manager.py`) ŌĆö eliminates all three classes at the root. Infrastructure is already in the repo: `VersionManager` is the worker's version-install engine, and `ui_api.validate_config` already uses it for pinned devices. This section generalizes that pattern to the default server path.

By default the server tracks the HA ESPHome add-on's version (already surfaced by `pypi_version_refresher._ha_addon_esphome_version`) with a fallback to a floor constant, so users' server-side scanner and `esphome config` validate always match the version they actually expect.

**First-boot UX:** 3ŌĆō5 minutes where the UI is functional but thin (no autocomplete, scanner metadata empty, validate returns 503). A top-of-app banner shows "Installing ESPHome X.Y.ZŌĆ”". Subsequent restarts reuse the installed venv ŌĆö no re-install.

## Firmware Download (pulled forward from 1.5)

Let users compile a target **without** the OTA flash step, then download the resulting firmware binary from the Queue tab. Two primary use cases: first-flash (USB flashing a new ESP that hasn't been OTA-adopted yet) and disaster recovery (device is unreachable but you still want the latest build in hand).

**Mode selection.** Chosen in the UpgradeModal via a third radio alongside the existing "Now" / "Scheduled" pair ŌĆö or, equivalently, a post-mode sub-toggle once the user picks "Now". Proposed UX: two radios inside "Now" mode labeled **"Compile + OTA"** (default, current behavior) and **"Compile + Download"**. When the second is chosen, the modal's Confirm button reads "Compile & Download" and the enqueued job carries a new `download_only: bool = true` flag. Scheduled mode always runs OTA ŌĆö download-only scheduled runs don't have a clear use case and can be re-evaluated later.

**Lifecycle.** Binary lifetime is tied to the **queue item**, not the target. The worker POSTs the compiled firmware back to the server; the server writes it to `/data/firmware/{job_id}.bin`; the server deletes the file when the user clears the job from the queue (individual Clear, bulk Clear Succeeded/Finished/All, or the existing per-target-coalescing cleanup in `enqueue()`). Retrying a downloaded-only job is a fresh compile ŌĆö the previous binary's storage is released when its job entry disappears. No TTL / age-based cleanup ŌĆö consistent with #18's "users clear explicitly" stance.

**Download surface.** A **Download** button appears on Queue-tab rows when `job.state == 'success' AND job.download_only AND job.has_firmware == true`. Clicking streams the file as `{target-stem}.bin` with appropriate `Content-Disposition`. No device-row entry point ŌĆö the concept only makes sense in a "this specific compile" context, which lives in the queue.
